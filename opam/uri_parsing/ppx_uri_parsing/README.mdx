# ppx_uri_parsing

Generates a `parser : t Uri_parsing.Parser.t`, alongside a `Ppx_uri_parsing_lib` module
that can be used to generate parsers for types that use `t`. This ppx aims to to supply
good defaults for people who don't care too much about all the details of their URIs.
If you want more power over your URIs, you should use the `Uri_parsing` library directly.

As a general rule of thumb, you should choose your types to match how you want your URI
to look, not the other way around.

## Basic examples

The basic ideas from URI parsing still hold: variants represent a choice of route, and
records represent a set of data. By default, the parser will try to read variant arguments
from the path and record arguments from the query.

Note that `typed_variants` or `typed_fields`, respectively, must be derived on the type
before `uri_parsing`.

<!-- $MDX file=doc/main_for_mdx.ml,part=simple-variant -->
```ocaml
module Admin_page = struct
  type t =
    | Settings
    | Edit_user of Id.t
  [@@deriving typed_variants, uri_parsing]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors Admin_page.parser;
  [%expect
    {|
    URL parser looks good!
    ┌─────────────────┐
    │ All urls        │
    ├─────────────────┤
    │ /edit-user/<id> │
    │ /settings       │
    └─────────────────┘
    |}]
;;
```

<!-- $MDX file=doc/main_for_mdx.ml,part=simple-record -->
```ocaml
module Search_params = struct
  type t =
    { query : string
    ; author_id : Id.t option
    ; categories : Id.t list
    }
  [@@deriving typed_fields, uri_parsing]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors Search_params.parser;
  [%expect
    {|
    URL parser looks good!
    ┌─────────────────────────────────────────────────────────────────────┐
    │ All urls                                                            │
    ├─────────────────────────────────────────────────────────────────────┤
    │ /?author-id=<optional<id>>&categories=<multiple<id>>&query=<string> │
    └─────────────────────────────────────────────────────────────────────┘
    |}]
;;
```

You can also derive `uri_parsing` on type aliases. Tuples act like records
with `first`, `second`, etc. fields, except that they read from the path by default.

<!-- $MDX file=doc/main_for_mdx.ml,part=simple-manifest -->
```ocaml
module User_data = struct
  type t = string * int [@@deriving uri_parsing]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors User_data.parser;
  [%expect
    {|
    URL parser looks good!
    ┌─────────────────┐
    │ All urls        │
    ├─────────────────┤
    │ /<string>/<int> │
    └─────────────────┘
    |}]
;;
```

By default, constructors and field names are translated to kebab-case, although this
can be overridden with the `~capitalize` argument (similar to `ppx_string_conv`):

<!-- $MDX file=doc/main_for_mdx.ml,part=capitalize -->
```ocaml
module T = struct
  type t = User_data of { user_id : Id.t }
  [@@deriving typed_variants, uri_parsing ~capitalize:"snake_case"]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors T.parser;
  [%expect
    {|
    URL parser looks good!
    ┌─────────────────────────┐
    │ All urls                │
    ├─────────────────────────┤
    │ /user_data?user_id=<id> │
    └─────────────────────────┘
    |}]
;;
```

## Attributes

Each of the attributes below are written in full form for documentation purposes
(e.g. `[@uri_parsing.sexpable]`) but can be abbreviated (e.g. `[@sexpable]`).
It's worth noting that `[@bool]` and `[@default]` in particular are used by `sexp`,
so you'll need explicit prefixing if you're deriving `sexp` and don't want to change
the sexp behavior.

### Sexpable, stringable, and binable values

By default, the ppx will try to use `Foo.Ppx_uri_parsing_lib.parser` to parse the type
`Foo.t`. `[@uri_parsing.sexpable]`, `[@uri_parsing.stringable]`, and
`[@uri_parsing.binable]` provide an "out" for values that don't have a parser.

<!-- $MDX file=doc/main_for_mdx.ml,part=sexpable -->
```ocaml
module My_id = struct
  type t = (Id.t[@uri_parsing.sexpable]) [@@deriving uri_parsing]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors My_id.parser;
  [%expect
    {|
    URL parser looks good!
    ┌─────────────┐
    │ All urls    │
    ├─────────────┤
    │ /<sexpable> │
    └─────────────┘
    |}]
;;
```

You can also generate these directly using the
`Ppx_uri_parsing_lib.Make_sexpable`, `Make_stringable`, and `Make_binable` functors:

<!-- $MDX skip -->
```ocaml
module My_id = struct
  type t = Id.t [@@deriving sexp]

  include functor Ppx_uri_parsing_lib.Make_sexpable
end
```

### Path vs. query vs. fragment

You can control whether a parser reads from the path or query by using the
`[@uri_parsing.path]` and `[@uri_parsing.query]` attributes. This won't always be
respected - variant routes can only be read from the path, for example, and options can
only be read from the query.

You can also specify that a value should be read from the fragment with
`[@uri_parsing.fragment]`. Unlike the path and query, only a single value can be read
from the fragment, and it must have an option type.

<!-- $MDX file=doc/path_query_for_mdx.ml,part=attrs -->
```ocaml
module Url = struct
  type t =
    | Homepage of { section : string option [@uri_parsing.fragment] }
    | Discussion of Id.t [@uri_parsing.query]
    | Search of
        { query : string
        ; author_id : Id.t option
        ; categories : Id.t list
        } [@uri_parsing.path]
  [@@deriving typed_variants, uri_parsing]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors Url.parser;
  [%expect
    {|
    URL parser looks good!
    ┌──────────────────────────────────────────────────────────┐
    │ All urls                                                 │
    ├──────────────────────────────────────────────────────────┤
    │ /discussion?value=<id>                                   │
    │ /homepage#<string>                                       │
    │ /search/<string>/<multiple<id>>?author-id=<optional<id>> │
    └──────────────────────────────────────────────────────────┘
    |}]
;;
```

### Variant routes

URIs often include a home page or a page with subpages. These pages can be marked with
`[@uri_parsing.index]` to indicate that there should be nothing further in the path.

You can also use `[@uri_parsing.route expr]` to control the route in a variant case
(defaults to the name of the constructor).

<!-- $MDX file=doc/routes_for_mdx.ml,part=attrs -->
```ocaml
module Url = struct
  type t =
    | Homepage [@uri_parsing.index]
    | Discussion of Id.t [@uri_parsing.route [ "forum" ]]
    | Search of { query : string } [@uri_parsing.route []]
  [@@deriving typed_variants, uri_parsing]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors Url.parser;
  [%expect
    {|
    URL parser looks good!
    ┌──────────────────┐
    │ All urls         │
    ├──────────────────┤
    │ /                │
    │ /?query=<string> │
    │ /forum/<id>      │
    └──────────────────┘
    |}]
;;
```

### Query defaults & flags

You can use `[@uri_parsing.default expr]` to indicate that, if this key is not present
in the query, that the field should parse to `expr`. If the field is equal to the default
(as determined by `[%equal t]`), it is dropped from the URI. Custom equality functions
are currently not supported.

You can also use `[@uri_parsing.bool]` to indicate a flag (i.e. a field that's true when
present in the query and false otherwise).

<!-- $MDX file=doc/query_defaults_for_mdx.ml,part=attrs -->
```ocaml
module Search_params = struct
  type t =
    { query : string [@uri_parsing.default ""]
    ; only_search_favorites : bool [@uri_parsing.bool]
    }
  [@@deriving typed_fields, uri_parsing]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors Search_params.parser;
  [%expect
    {|
    URL parser looks good!
    ┌────────────────────────────────────────────────────┐
    │ All urls                                           │
    ├────────────────────────────────────────────────────┤
    │ /?[only-search-favorites]&query=<optional<string>> │
    └────────────────────────────────────────────────────┘
    |}]
;;
```

### Fallbacks

While defaults help when a key is not present in the query, it doesn't handle when a
value is present but fails to parse. This is what `[@uri_parsing.fallback expr]` does.
It can handle single values in either the path or query.

<!-- $MDX file=doc/fallback_for_mdx.ml,part=attrs -->
```ocaml
module Admin_page = struct
  type t =
    | Settings
    | Edit_user of Id.t [@uri_parsing.fallback 0]
  [@@deriving typed_variants, uri_parsing]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors Admin_page.parser;
  [%expect
    {|
    URL parser looks good!
    ┌───────────────────────────┐
    │ All urls                  │
    ├───────────────────────────┤
    │ /edit-user/<fallback<id>> │
    │ /settings                 │
    └───────────────────────────┘
    |}]
;;
```

### Namespaces & keys

Namespaces control the names of query keys - for example, your `a` field might actually
appear as `foo.a` if it's nested inside a `foo` field.

`ppx_uri_parsing` automatically adds record field names to the namespace, but not
variant constructor names. You can customize this behavior by attaching
`[@uri_parsing.key expr]` with your desired namespace/key name, or `[@uri_parsing.no_key]`
if you don't want to add anything to the namespace. When the lowest-level namespace is
empty, `value` is used as a substitute key name.

This can be useful if you want to use a record to combine two different parsers without
anything extra appearing in the namespace, if your variant contains query arguments
whose names would conflict with an inner variant, or if you simply want a different
key name.

<!-- $MDX file=doc/main_for_mdx.ml,part=key -->
```ocaml
module Contrived_key_example = struct
  type t =
    | Default of { a : int * int }
    | Keep_both_keys of { a : int * int } [@uri_parsing.key "kept"]
    | Skip_both_keys of { a : int * int [@uri_parsing.no_key] }
    | Keep_outer_skip_inner of { a : int * int [@uri_parsing.no_key] }
    [@uri_parsing.key "kept"]
  [@@deriving typed_variants, uri_parsing]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors Contrived_key_example.parser;
  [%expect
    {|
    URL parser looks good!
    ┌───────────────────────────────────────────────────────────┐
    │ All urls                                                  │
    ├───────────────────────────────────────────────────────────┤
    │ /default?a.first=<int>&a.second=<int>                     │
    │ /keep-both-keys?kept.a.first=<int>&kept.a.second=<int>    │
    │ /keep-outer-skip-inner?kept.first=<int>&kept.second=<int> │
    │ /skip-both-keys?first=<int>&second=<int>                  │
    └───────────────────────────────────────────────────────────┘
    |}]
;;
```

### Custom parsers

If these features cannot achieve the parser you desire, you can create a parser by hand.

If you want to define a custom parser for a whole type, consider using the
`Ppx_uri_parsing_lib.Make_from_value_parser` or `Make_from_parser` functor:
<!-- $MDX skip -->
```ocaml
module Id = struct
  type t = int [@@deriving sexp]

  let value_parser = Uri_parsing.Value_parser.name "id" Uri_parsing.Value_parser.int

  include functor Ppx_uri_parsing_lib.Make_from_value_parser
end
```

Alternatively, you could use the `[@uri_parsing.custom_parser expr]` attribute.
This expression must have the same type that the ppx uses internally to represent parsers
(`(t, _) Ppx_uri_parsing_lib.Derived_parser.t`). You can use existing parsers in the
`Ppx_uri_parsing_lib` library to construct your parser, or you can construct it from scratch:

<!-- $MDX file=doc/custom_for_mdx.ml,part=custom -->
```ocaml
let parser_for_id ~parse_from ~namespace:_ =
  Parser_with_kind.Value_parser
    (Uri_parsing.Value_parser.name "id" Uri_parsing.Value_parser.int, parse_from)
;;

module Search_params = struct
  type t =
    { query : string
    ; author : Id.t option [@uri_parsing.custom_parser parser_for_option parser_for_id]
    ; categories : (Id.t[@uri_parsing.custom_parser parser_for_id]) list
    }
  [@@deriving typed_fields, uri_parsing]
end

let%expect_test _ =
  Uri_parsing.Parser.check_ok_and_print_urls_or_errors Search_params.parser;
  [%expect
    {|
    URL parser looks good!
    ┌──────────────────────────────────────────────────────────────────┐
    │ All urls                                                         │
    ├──────────────────────────────────────────────────────────────────┤
    │ /?author=<optional<id>>&categories=<multiple<id>>&query=<string> │
    └──────────────────────────────────────────────────────────────────┘
    |}]
;;
```

## Parser composition

Normally, `ppx_uri_parsing` derives a signature that looks something like

```ocaml skip
type t [@@deriving uri_parsing]

(* Generates... *)
val parser : t Uri_parsing.Parser.t

module Ppx_uri_parsing_lib : sig
  val parser : (t, [ `Parser ]) Ppx_uri_parsing_lib.Derived_parser.t
end
```

The first `parser` is intended for people to use directly; the second is for the ppx to
use in deriving further parsers.

However, types with type parameters can't be used directly, since they're missing parsers
corresponding to those parameters. In this case, only the `Ppx_uri_parsing_lib` module is
generated, and its parser takes in an extra parser argument for each type parameter.

```ocaml skip
type 'a t [@@deriving uri_parsing]

(* Generates... *)
module Ppx_uri_parsing_lib : sig
  val parser
   : ('a, _) Ppx_uri_parsing_lib.Derived_parser.t
  -> ('a t, [ `Parser ]) Ppx_uri_parsing_lib.Derived_parser.t
end
```

In turn, deriving `uri_parsing` on something like `type t' = int t` will generate a
top-level `parser_for_t'`.

Some types (e.g. options and lists) can only use value parsers as arguments. In this
case, you will need to manually write out the signature in the MLI and indicate which
arguments need to be value parsers.

```ocaml skip
val parser_for_option
 : ('a, [ `Value_parser ]) Ppx_uri_parsing_lib.Derived_parser.t
-> ('a t, [ `Parser ]) Ppx_uri_parsing_lib.Derived_parser.t
```

If someone passes a regular parser as an argument (or has something like
`(int * bool) option` in their type), they will get a type error saying that
`[ ``Value_parser ]` and `[ ``Parser ]` do not match.

## Migration cheat sheet

Most unsupported constructs below can be inserted into `ppx_uri_parsing` with
`[@uri_parsing.custom_parser expr]`; however, it can be cumbersome to write out an
underlying, derived parser. If you want any of the below marked "Currently not supported"
in the ppx, reach out to us and we'll see if we can add it.

```
-----------------------------------------------------------------------------------------
| Uri_parsing construct                   | ppx_uri_parsing equivalent                  |
-----------------------------------------------------------------------------------------
| Value_parser.int                        | int                                         |
| Value_parser.string                     | string                                      |
| Value_parser.time_ns                    | Currently not supported                     |
| Value_parser.float                      | float                                       |
| Value_parser.bool                       | bool                                        |
| Value_parser.stringable                 | [@uri_parsing.stringable]                   |
| Value_parser.sexpable                   | [@uri_parsing.sexpable]                     |
| Value_parser.binable_via_base64         | [@uri_parsing.binable]                      |
| Value_parser.base64_encoded             | Currently not supported                     |
| Value_parser.project                    | Currently not supported                     |
| Value_parser.fallback                   | [@uri_parsing.fallback expr]                |
| Value_parser.name                       | Currently not supported                     |
-----------------------------------------------------------------------------------------
| Parser.project                          | Currently not supported                     |
| Parser.optional_query_fields            | Currently not supported                     |
| Parser.from_fragment                    | [@uri_parsing.fragment]                     |
| Parser.from_query_required              | Record fields or [@uri_parsing.query]       |
| Parser.from_query_optional              | t option                                    |
| Parser.from_query_flag                  | [@uri_parsing.bool]                         |
| Parser.from_query_optional_with_default | [@uri_parsing.default expr]                 |
| Parser.from_query_many                  | t list                                      |
| Parser.from_query_many_at_least_1       | Currently not supported                     |
| Parser.unit                             | Variant with no arguments                   |
| Parser.from_path                        | Variant arguments or [@uri_parsing.path]    |
| Parser.from_remaining_path              | t list [@uri_parsing.path]                  |
| Parser.with_prefix                      | Variant constructors,                       |
|                                         |   [@uri_parsing.route expr]                 |
| Parser.with_remaining_path              | Nesting [@uri_parsing.index] variant inside |
|                                         |   a [@uri_parsing.route expr] variant       |
| Parser.end_of_path                      | [@uri_parsing.index]                        |
| Parser.both                             | t1 * t2                                     |
| Parser.new_parser                       | Currently not supported                     |
-----------------------------------------------------------------------------------------
| Variant.make                            | variant type                                |
| Query_based_variant.make                | Currently not supported                     |
| Record.make                             | record type                                 |
-----------------------------------------------------------------------------------------
| Versioned_parser                        | Not supported; do this manually             |
-----------------------------------------------------------------------------------------
| ~namespace or ~key argument             | [@uri_parsing.key expr] or                  |
|                                         |   [@uri_parsing.no_key]                     |
-----------------------------------------------------------------------------------------
```
